\subsection*{Phase 2: The Security "Great Wall" (Bugs \& Solutions)}
This was my first major hurdle. I had to configure the Security Groups to allow the nodes to talk.
\begin{itemize}
    \item \textbf{The Bug}: The instances couldn't "see" each other initially because AWS blocks all traffic by default.
    \item \textbf{The Struggle}: I encountered an error: "You may not specify a referenced group id for an existing IPv4 CIDR rule."
    \item \textbf{The Fix}: I learned that I cannot mix CIDR (0.0.0.0/0) with Security Group IDs in one rule. I created a Self-Referencing Rule for \texttt{sg-09707f49c49aa2a50} to allow All TCP traffic internally.
\end{itemize}

\subsection*{Phase 3: The SSH Authentication Battle}
This is where I had to do the most troubleshooting.
\begin{itemize}
    \item \textbf{The Permission Bug}: My local laptop rejected the \texttt{.pem} key because permissions were "too open" (0664). \\
    \textit{Solution}: I used \texttt{chmod 400 taha-key.pem} to lock it down.
    \item \textbf{The "Permission Denied (publickey)" Bug}: When I tried to link the Master to the Slave, AWS blocked the connection because \textit{PasswordAuthentication} was disabled. 
    \item \textbf{The Discovery}: I found that Ubuntu 24.04 on AWS has a "double lock." Even after changing the main \texttt{sshd\_config}, a hidden file \texttt{60-cloudimg-settings.conf} was overriding my settings.
    \item \textbf{The Fix}: I edited the cloud-init settings on the Slave, restarted the SSH service, and successfully performed the SSH Handshake (\texttt{ssh-copy-id}).
\end{itemize}

\subsection*{Phase 4: Hadoop Configuration (The Brain)}
I customized the XML files to move from a "Local" mode to a "Distributed" mode.
\begin{itemize}
    \item \textbf{The Setup}: I created dedicated directories (\texttt{tmpdata, dfsdata}) under \texttt{/home/hdoop/} to ensure the system had write permissions.
    \item \textbf{The Logic}: I mapped the Private IPs in \texttt{/etc/hosts} so the cluster uses names (\texttt{hadoop-master, hadoop-slave}) instead of manually changing IPs.
    \item \textbf{Synchronization}: I used \texttt{scp} to "push" the Master’s configuration to the Slave, ensuring the cluster was perfectly synced.
\end{itemize}

\subsection*{File Properties Questions}

\begin{enumerate}
    \item \textbf{What is the default block size (in Mb) of the test.txt file?} \\
    The default block size is 128 MB. In Hadoop 3.x, the default \texttt{dfs.blocksize} is 134,217,728 bytes. Even though the \texttt{test.txt} file is small (122 bytes), HDFS allocates it into one full 128MB block metadata entry.
    \item \textbf{Is there any missing replicas for the file test.txt?} \\
    No, there are 0 missing replicas. The \texttt{hdfs fsck} report showed "Missing replicas: 0 (0.0 \%)". Since the DataNode is live on the local system, the single required replica was successfully stored.
    \item \textbf{What command will you use to change this block size? (remember to convert into bytes)} \\
    Use the \texttt{-Ddfs.blocksize} flag during the put command:\\
    \texttt{hdfs dfs -Ddfs.blocksize=1048576 -put test.txt /test\_small.txt}\\
    (Note: 1048576 bytes = 1 MB).
    \item \textbf{How many blocks are used by test.txt after changing block size above?} \\
    1 Block. Since the file size (122 bytes) is much smaller than the new 1MB (1,048,576 bytes) limit, it still fits within a single block.
    \item \textbf{How many missing replicas are there for file test.txt after block change?} \\
    0 Missing Replicas. As long as the single DataNode is running, the file is considered healthy with a replication factor of 1.
    \item \textbf{Why are there missing replicas?} \\
    In a Pseudo-Distributed/Single-Node setup, there is only one DataNode. If the replication factor is set to 3, Hadoop tries to find 3 different nodes. Since it only finds one, it reports the other 2 as "missing" because it cannot provide the requested level of redundancy.
\end{enumerate}
